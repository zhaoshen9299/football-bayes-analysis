# 足球贝叶斯分析

## 工作原则

把分析当作带时间戳的概率预测，而不是赛果断言。

- 区分四类内容：外部事实、模型输入、模型派生结果、尚未验证的战术假设。
- 把“贝叶斯”落实为先验、似然、后验与后验预测；不要用主观百分点加减冒充更新。
- 先建立市场去水基准，再判断模型是否提供增量信息。没有回测支持时，不把模型与市场任意混合。
- 用区间、情景与数据等级表达不确定性。数据弱时降低结论强度，不制造小数点精度。
- 只做赛前分析。若比赛已经开始，说明本流程不适用，并询问是否改做赛后复盘。
- 概率分析不等于投注指令；不得承诺命中、盈利或“稳胆”。

## 按需读取资源

- 需要搜集或判断数据时，读取 [references/data-sources.md](references/data-sources.md)。
- 需要选择模型、计算更新、处理战术修正或解释指标时，读取 [references/methodology.md](references/methodology.md)。
- 需要运行确定性计算时，读取 [references/input-contract.md](references/input-contract.md)，再运行 `scripts/predict_match.py`。
- 需要生成最终报告时，读取 [references/report-template.md](references/report-template.md)。
- 需要赛后评估一批冻结预测时，运行 `scripts/evaluate_forecasts.py`；指标定义见方法参考。

## 输入解析

从用户文本中提取：赛事、主队、客队、开球时间及时区、场地、是否中立、是否两回合、用户关心的市场、分析截止时间。

按以下规则处理：

1. 优先通过官方赛程消除队名、赛事与时间歧义。
2. 仅当歧义会改变比赛身份或模型语境且无法查证时，提出一个简短问题。
3. 记录规范比赛键：`赛事|赛季|开球UTC|主队|客队`。刷新分析时沿用同一比赛键并生成新快照。
4. 杯赛、淘汰赛、中立场、加时或两回合必须单独标记。默认胜平负仅指常规时间；不得把晋级概率混入 90 分钟概率。
5. 用户未指定输出深度时，默认输出“标准版”；多场任务按数据可得性分批，并给出已完成、待处理和快照时间。

## 建立赛前快照

先固定分析截止时间，避免使用截止时间之后的信息。对每项关键输入记录：

| 字段 | 要求 |
|---|---|
| 值或结论 | 使用原始口径，不先改写 |
| 状态 | `CONFIRMED`、`REPORTED`、`INFERRED` 或 `ASSUMED` |
| 来源 | 直接页面或数据端点 |
| 事件时间 | 信息发生或发布的时间 |
| 获取时间 | 本次抓取时间与时区 |
| 数据供应商 | 尤其标明 xG、赔率和阵容供应商 |
| 置信等级 | A、B、C，或无法使用 |

引用外部数字时给出来源；引用派生概率时给出模型、输入快照和计算方式。不要展示冗长的内部工具日志。

## 数据获取顺序

按 [references/data-sources.md](references/data-sources.md) 执行，并至少覆盖以下维度：

1. 确认赛程、场地、赛事规则与比赛状态。
2. 获取可比的历史结果或同供应商 xG 数据，保留主客场、日期、对手和样本口径。
3. 获取停赛、伤情、预计首发；临近开赛时区分预计与官方首发。
4. 获取同一时间窗口内、至少两家机构的十进制胜平负赔率；只有一家时明确单源风险。
5. 获取赛程密度、旅行、换帅、天气或裁判等信息，但仅在有机制和可靠证据时纳入。
6. 重复查询时重新抓取时间敏感字段，并列出与上一个快照的变化；不得直接复用旧伤停或旧赔率。

不要混合不同供应商的 xG 序列来制造“近期趋势”。不要把球队总控球率、排名或单场比分直接当作实力。

## 评定数据层级

在计算前选择并公开一个层级：

- **L1 完整模型**：有同口径历史比赛级进球或 xG、联赛基线、时间信息、可靠阵容信息和多机构赔率。可运行动态分层进球模型或已验证的外部模型。
- **L2 收缩模型**：有同口径近期进球或 xG、联赛主客场基线和赔率，但不足以拟合完整层级模型。使用本 Skill 的经验贝叶斯 Gamma-Poisson 近似。
- **L3 市场基准**：历史表现数据不可靠，但有带时间戳的多机构赔率。只输出去水市场概率与定性情景，不称为独立模型优势。
- **L4 不足**：连可靠赔率或可比历史数据也没有。将数字概率标记为 `UNAVAILABLE`，只报告已确认事实和缺口。

层级由数据决定，不因用户想要“更深”而升级。

## 计算流程

### 1. 建立市场基准

逐家机构把十进制赔率转成隐含概率并去水，再对各机构公平概率取中位数并归一化。保留赔率时间戳和中位返还率/水位。使用 `scripts/predict_match.py` 计算，避免手算错误。

### 2. 建立进球率后验

- L1：优先使用经过时间切分验证的动态分层 Poisson、双变量 Poisson或 Dixon-Coles 模型。
- L2：使用 `scripts/predict_match.py` 的 `empirical_bayes` 模式，对主队主场进攻、主队主场防守、客队客场进攻、客队客场防守分别做时间衰减和联赛均值收缩。
- 已有外部模型的后验进球率时，使用 `direct_rates` 模式传播不确定性。
- L3：使用 `market_only` 模式，不反推出虚假的球队进球率。

仅在对对应赛事做过时序回测时使用非零 Dixon-Coles `rho`。否则设为 0 并说明。

### 3. 处理阵容与战术信息

把战术判断写成可证伪机制，例如“客队高位逼抢可能增加主队后场失误”，再检查支持与反证。不得使用“控球率 > 60% 且 Field Tilt < 50%”之类跨联赛硬阈值。

只有存在经历史估计、球员模型或明确先验的影响映射时，才把信息写成进球率对数乘数。否则：

1. 保持基础后验不变；
2. 建立具名情景；
3. 给出该情景下的敏感性区间；
4. 不把情景结果伪装成无条件后验。

伤停的状态必须区分已确认缺阵、出战存疑和媒体预计。不要按球员名气直接加减胜率。

### 4. 生成后验预测

从后验进球率抽样，构造比分矩阵，再汇总：

- 90 分钟胜、平、负概率；
- 后验区间；
- 最可能比分，但不把最高概率比分称作“预测比分”；
- 用户要求时才输出大小球、双方进球或让球相关结果；
- 模型相对去水市场的百分点差异。

检查概率非负且胜平负合计为 100%。报告显示通常保留 1 位小数；内部计算保留更高精度。

### 5. 判断是否存在可表达的差异

不要仅凭点估计高于市场就称为“价值”。至少同时检查：

- 模型在历史时序回测中是否优于同时间点市场基准；
- 当前差异是否大于模型误差、数据延迟和赔率分散；
- 结论是否对阵容或进球率的小幅变化敏感；
- 赔率是否仍是报告中记录的价格。

若这些条件不满足，使用“模型与市场存在方向性差异，但证据不足”这一层级。

## 输出报告

使用 [references/report-template.md](references/report-template.md)，并遵循以下顺序：

1. 先给比赛身份、截止时间、数据层级与一句结论。
2. 并排给出模型概率、90% 区间和去水市场概率；没有模型时明确留空。
3. 解释从先验/基础后验到当前后验的证据链，不输出无法复现的“+3%”。
4. 给出 2 个有触发条件的比赛情景；情景概率只在有模型依据时提供。
5. 给出主要不确定性、会使结论失效的新信息和下一次建议刷新时间。
6. 紧邻事实与数字放置来源链接；末尾列出最关键来源即可。

不要强制插入与本场无关的“无效控球历史反例”。只有它能验证本场机制时才引用，并确保同口径可追溯。

## 刷新、多场与断点续写

- 每次刷新创建新快照，保留旧概率，不覆盖历史值。
- 输出变化表：字段、旧值、新值、来源、变化时间、对模型的影响。
- 只重算受变化影响的模型；代码或参数改变时必须标注“方法变化”，不得把它写成新闻导致的概率变化。
- 多场批处理共享数据规则，不共享球队参数、阵容假设或陈旧赔率。
- 分批结束时输出检查点：比赛键、已完成状态、最后快照、未解决缺口。

## 赛后校准

把开赛前冻结的预测按 JSONL 保存，赛果产生后运行：

```bash
python scripts/evaluate_forecasts.py forecasts.jsonl
```

使用时间顺序评估，至少报告 Log Loss、Multiclass Brier、校准分箱及相同截止时间的市场基准。样本量不足时不下强结论。不得用赛果之后的数据回填赛前预测。

## 禁止事项

- 不编造伤停、首发、赔率、xG、Field Tilt、Elo 或概率来源。
- 不把搜索摘要、聚合站传闻或同名球队页面直接当作已确认事实。
- 不把不同供应商、不同定义或不同赛季口径的高级指标直接拼接。
- 不根据单场控球、射门或比分做大幅实力更新。
- 不声称盘口变化的原因，除非有直接报道；只能描述变化与可能机制。
- 不把最高概率结果写成“必然”，不输出保证收益、追损或确定性投注语言。
- 不用“20 年经验专家”等不可验证身份包装结论。
